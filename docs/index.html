Objective

Design and characterise an affordable, extensible environmental monitoring platform for observing fungal growth through a solid substrate. The system logs temperature, relative humidity, atmospheric pressure, volatile organic compounds (VOC proxy via gas resistance), and spectral light intensity every 3 seconds, streaming data from an Arduino over serial to a host PC. The goal is to find sensor-based proxies for substrate colonization — using ambient-vs-in-bag differentials in temperature, humidity, and gas resistance as indicators of fungal metabolic activity, and light transmission through the bag as an indicator of mycelial density.

Study 001 (pilot) began 2026-06-30. A liquid culture of fungus was inoculated into a bag of sterile substrate and outfitted with sensors the same day at 5:00pm. Sensor placement is critical to interpretation: SHT30 and BME688 #1 sit outside the bag, on the table next to it, and measure ambient room conditions. BME688 #2 sits inside the bag, in direct contact with the substrate headspace. The AS7341 light sensor sits under the bag, illuminated from above by a light strong enough to saturate the sensor (1000 across all channels) if the bag were not attenuating it — so light transmitted through the bag, not absolute brightness, is the signal of interest.

Platform code, raw data, and this analysis dashboard are version-controlled in this repository.

Hardware

Arduino microcontroller (no on-board RTC) streams comma-separated sensor rows at 115,200 baud. All sensors share a single I²C bus. Absolute wall-clock timestamps are assigned by the host PC logger; the Arduino provides only millis()-based uptime. Two BME688 sensors occupy the same I²C bus via their address-select pin (SDO): BME688 #1 has SDO pulled to GND → address 0x76; BME688 #2 has SDO pulled to 3.3 V → address 0x77.

Study 001 sensor placement — key interpretation context

Sensor position is fixed for study 001 and directly affects how every chart should be read. This placement must be documented here because it is not encoded in the CSV data:

■ SHT30 (0x44)
On the table, outside the bag. Measures ambient room temperature and relative humidity. Capacitive polymer RH element; accuracy ±0.2 °C, ±2 % RH (typ.). Not in contact with the substrate.
■ BME688 #1 (0x76)
On the table, outside the bag, next to SHT30. Measures ambient T, RH, P, and background lab VOCs. Provides an environmental baseline and drift reference for BME688 #2. MOX sensor requires 1–3 h burn-in per session.
■ BME688 #2 (0x77)
Positioned inside the substrate bag, in the headspace above the substrate. Temperature and humidity reflect in-bag microclimate, not room conditions. Gas resistance tracks VOCs building up in the enclosed headspace, potentially including fungal metabolic off-gassing. This is the sensor most directly exposed to the fungal culture.
■ AS7341 (0x39)
DFRobot Gravity module. The AS7341 chip has 8 narrowband channels (F1–F8, 415–680 nm) + CLEAR + NIR, but reading all 8 requires two separate measurement passes (DFRobot_AS7341 Mode 1 and Mode 2). Study 001's firmware only runs Mode 1 (startMeasure(eF1F4ClearNIR) / readSpectralDataOne()) — so only F1–F4 + CLEAR + NIR (6 channels) are actually logged; F5–F8 were never captured. Positioned under the bag, beneath a light source strong enough to saturate the sensor if unobstructed. Intended as a mycelial-colonization proxy: as the fungus colonizes the substrate and the bag interior fills with white mycelium, transmitted light reaching the sensor is expected to fall.

Wiring

All four sensors share a single I²C bus (Arduino SDA/SCL — pins A4/A5 on an Uno-class board). SHT30 and the AS7341 have fixed I²C addresses; the two BME688 sensors are otherwise identical modules distinguished only by the state of their SDO (address-select) pin — SDO tied to GND gives address 0x76 (BME688 #1), SDO tied to VCC gives 0x77 (BME688 #2). The pump and stepper are driven directly by digital/PWM output pins and are not on the I²C bus.

Source: user_provided/arduino/01_stepper_motor/addresses.txt and the pin definitions in 01_stepper_motor.ino.

Software

Arduino Firmware

user_provided/arduino/01_stepper_motor/01_stepper_motor.ino — C/C++. Reads all sensors every 3 seconds, streams 22 comma-separated fields over serial at 115,200 baud, and manages peristaltic pump duty-cycle (0 min ON / 1000 min OFF by default — effectively disabled for Study 001) and stepper motor speed. No RTC — uptime is millis() since boot.

Required Arduino libraries:

Wire (built-in) Adafruit_SHT31 Adafruit_BME680 DFRobot_AS7341

Python Data Loggers

Active scripts live in user_provided/arduino/01_stepper_motor/; archived copies in user_provided/python/archive/. Each script opens the Arduino serial port, rejects lines without exactly 22 fields (handles startup noise and truncated packets), prepends pc_timestamp and elapsed_s, and appends rows to a timestamped CSV in the correct studies/ subfolder.

Output path is resolved automatically — three levels up from the script file gives the repo root, then studies/STUDY_NAME/. Set STUDY_NAME and SERIAL_PORT at the top of the active script before each session. On Linux the port is typically /dev/ttyUSB0; on Windows COM3/COM4.

Data Pipeline

user_provided/python/generate_study_summaries.py — full pipeline script. Run via bash user_provided/makefile/build.sh which also opens the site. Key internal design decisions for future reference:

Schema normalisation: Five historical CSV versions (A–E) exist from the prior project on this rig, as sensors were added incrementally. A COL_ALIASES dict in the script maps every legacy column name to a canonical name (e.g., temp_C → sht30_temp_C). All versions are processed identically after normalisation.

Bad-data filtering applies three classifications — nan, below_min, above_max — using per-column limits in SENSOR_LIMITS: temperature 0–80 °C, humidity 0–100 %, pressure 800–1,100 hPa, gas resistance 100–500,000,000 Ω. Filtered values are replaced with None (→ JSON null → Plotly line break).

Downsampling: step = max(1, N // MAX_CHART_POINTS) where MAX_CHART_POINTS = 4000. Gaps between sessions longer than 60 s receive a null-row separator so Plotly does not draw a connecting line across the gap.

description.txt: Place a plain-text file at studies/study00X_pilot/description.txt — it is read by read_description() and shown verbatim at the top of the study block on the site.

Auto-interpretation: generate_interpretation() classifies temperature stability (range <2 °C / <5 °C / >5 °C), inter-sensor agreement (BME688 #1 vs #2 ΔT, SHT30 vs BME688 offset), gas resistance level and fractional variation, and bad-data fraction, producing a temp_pressure_text string and a list of bullets rendered below each study.

Trend, Diurnal & Model-Fit Regression Analysis

Each study page shows a per-variable trend table (labelled "Table 2" on the study page) with five columns: slope, Pearson r, a 24-hour diurnal-fit verdict, and a best-fitting functional form. All of it is computed once, at build time, in user_provided/python/generate_study_summaries.py — functions ols(), diurnal_fit_r2(), best_fit_model(), and compute_trend_stats() — using only valid (non-bad) readings and the Python standard library (no numpy/scipy). This section documents the method in full so results can be reproduced or checked by hand.

Time basis. Every fit below uses each sample's absolute epoch time, converted to hours (dt.timestamp() / 3600), as the independent variable. This matters for two reasons: (1) slope and Pearson r are translation-invariant, so the result does not depend on an arbitrary choice of "time zero"; and (2) using true epoch time keeps the diurnal fit's phase tied to actual time-of-day rather than to each variable's own first sample.

1  Linear-regression slope & r

Ordinary least squares fit of value y against elapsed time t (hours):

slope = Σ(tᵢ − t̄)(yᵢ − ȳ) / Σ(tᵢ − t̄)²        [unit: variable-unit per hour]

r = Σ(tᵢ − t̄)(yᵢ − ȳ) / √( Σ(tᵢ − t̄)² · Σ(yᵢ − ȳ)² )   [Pearson correlation, −1 to 1]

Slope is the average rate of change over the whole study; a positive slope means the variable trended upward, negative means downward. r measures how well a straight line describes the data — |r| near 1 means the trend is close to linear and consistent; |r| near 0 means either no trend or a trend that isn't linear (see the model-fit column for the latter case). Requires at least 3 valid points; shown as "—" otherwise.

2  24-hour diurnal-cycle fit

A single 24-hour sinusoid is fit to the data by closed-form least squares (three-parameter normal equations, solved by Gaussian elimination):

θᵢ = 2π · tᵢ / 24                         [t in hours → θ has a 24h period]

y = a + b·sin(θ) + c·cos(θ)                [fit a, b, c by least squares]

R²_diurnal = 1 − SS_res / SS_tot            [fraction of variance explained]

Fitting a sin/cos pair rather than a single sinusoid lets the regression find whatever phase fits best (any amplitude/phase combination is a linear combination of sin and cos), so R²_diurnal does not depend on the arbitrary time origin. The table reports "Yes" when R²_diurnal ≥ 0.15 (at least 15% of the variable's variance is explained by a 24h cycle) and "No" otherwise. This threshold is a heuristic amplitude screen, not a formal significance/p-value test — it is not corrected for multiple comparisons across variables, and a short study (Study 001 ran ~19 h, less than one full day) has limited statistical power to confirm a true 24h period versus a monotonic drift that merely resembles one over a partial cycle. Requires at least 20 valid points; shown as "—" otherwise.

3  Best-fit functional form

Four candidate models are fit to the same (t, y) data and compared on equal footing. Three of the four are nonlinear in their raw parameters, so each is linearised by a log transform before applying the same closed-form OLS as above, then the fitted curve is evaluated back in original y-units for a fair R² comparison:

Linear:        y = a + b·t                    fit directly by OLS on (t, y)

Exponential:   y = a·exp(b·t)                  fit OLS on (t, ln y);  a = exp(intercept)
               "Growth" if b > 0, "Decay" if b < 0.  Requires y > 0 everywhere.

Logarithmic:   y = a + b·ln(t)                 fit OLS on (ln t, y)
               t is shifted so t > 0 (t' = t − min(t) + 0.01h) — ln is undefined at t=0.

Power law:     y = a·t^b                       fit OLS on (ln t, ln y);  a = exp(intercept)
               Requires y > 0 everywhere (uses the same shifted t' as above).

R² for every candidate is computed as 1 − SS_res/SS_tot using ŷ back in original
y-units — never in the transformed (ln) space — so all four models are compared
on the same scale.

The candidate with the highest R² is reported only if that R² is at least 0.30 (explains ≥30% of variance); otherwise the cell reads "None" — meaning no simple functional form (of the four tried) fits the data meaningfully better than noise. A "Linear" result here is a stronger claim than the slope/r columns alone: it means the linear form specifically outperformed the exponential, logarithmic, and power-law alternatives, not merely that r is close to ±1. Requires at least 3 valid points.

Caveat: with only four fixed candidate shapes and no cross-validation or corrected degrees of freedom, "best fit" here means best among these four forms, not best possible model — treat it as a descriptive summary of the dominant shape in the data, not a confirmed mechanism.

Sensor Physics & Substrate Bag Thermodynamics

The sensors used in this system measure variables that are physically coupled through well-established relationships. Understanding these relationships is essential for separating true biological signals from sensor artefacts, environmental drift, and placement effects.

1  Ideal Gas Law — Temperature–Pressure Coupling

For an ideal gas in a closed, constant-volume vessel:

PV = nRT          [SI units: Pa, m³, mol, J·mol⁻¹·K⁻¹]

Constant n, V:    P₁/T₁ = P₂/T₂   →   dP/dT = P/T ≈ 3.4 hPa/K at STP

The substrate bag is not a sealed rigid vessel, so temperature and pressure inside it are not tightly coupled the way the ideal-gas relation above would predict for a closed system: any flexing or venting of the bag lets internal pressure track the surrounding atmosphere. Any apparent T–P correlation between BME688 #1 (outside) and BME688 #2 (inside) most likely reflects meteorological pressure variation or a BME688 temperature-compensation artefact, not fungal activity — pressure is a poor proxy for what is happening inside the bag.

2  Saturation Vapour Pressure — Antoine Equation

The saturation vapour pressure of water P* as a function of temperature T:

log₁₀(P*) = A − B / (C + T)     [T in °C, P* in mmHg → ×1.33322 for hPa]

Antoine constants for water (1–100 °C; NIST Webbook):
  A = 8.07131,  B = 1730.63,  C = 233.426

Key values:
  P*(22 °C) ≈ 26.4 hPa   (typical ambient room temperature)
  P*(30 °C) ≈ 42.4 hPa   (representative in-bag temperature)
  Ratio: 1.6 — a warmer bag interior can hold ~60% more water vapour before saturating

Clausius-Clapeyron slope (liquid–vapour equilibrium):
  dP*/dT = Lᵥ · P* / (Rᵥ · T²)
  Lᵥ = 2.501 × 10⁶ J/kg  (latent heat of vaporisation)
  Rᵥ = 461.5 J/(kg·K)    (specific gas constant for water)
  At T = 300 K: dP*/dT ≈ 1.8 hPa/°C

This means BME688 #2, sealed inside the substrate bag with a moist substrate, is expected to read substantially higher relative humidity than the bench sensors outside — this is normal for an enclosed, moist microclimate and does not by itself indicate fungal activity. Humidity readings at or near 100 % RH may reflect condensation on the sensor element rather than an accurate reading.

3  BME688 Metal-Oxide Gas Sensor — Response Model

The BME688 uses a heated metal-oxide (MOX) sensing layer whose resistance changes in the presence of reducing gases:

R = R₀ · [CVOC]⁻ᵅ · exp(Eₐ / kT)

Parameters:
  R₀   — baseline resistance in clean air [Ω]
  α    ≈ 0.3–0.7   (power-law exponent; gas-dependent)
  Eₐ   ≈ 0.1–0.5 eV (activation energy for surface reaction)
  k    = 8.617 × 10⁻⁵ eV/K (Boltzmann constant)
  T    — sensor hot-plate temperature (~340 °C operational)

Sensitivity: dR/d[C] = −α · R₀ · [C]⁻⁽ᵅ⁺¹⁾   (compresses at high [C])

Fungal metabolism is a known source of reducing VOCs (e.g. ethanol and other
alcohols, aldehydes) — the specific compound mix depends heavily on species and
growth stage and is not characterised for this study. The sensor's response is a
non-selective aggregate signal, not a speciated measurement.

CO₂ is NOT detected by n-type MOX sensors (fully oxidised; no electron donation),
so respiration-driven CO₂ buildup in the bag headspace is invisible to this sensor.

The sensor is non-selective: R integrates the response to all reducing gases simultaneously and cannot on its own distinguish a fungal VOC from ambient lab solvents or cleaning agents. Discrimination between sources would require temperature cycling or multi-sensor arrays (e-nose approach). Burn-in time of 1–3 h per session is required before R₀ stabilises — early-session gas readings should be treated cautiously.
